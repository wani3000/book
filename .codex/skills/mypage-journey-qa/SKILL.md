# My Page Journey QA

## 목적

마이페이지가 하나의 긴 화면에 섹션을 숨겨 놓은 형태가 아니라, 사용자가 예측할 수 있는 독립 페이지와 브라우저 이동 규칙을 갖추었는지 검증한다.

## 역할 원본

작업 시작 전에 `.codex/harness/agents/mypage-journey-qa.md`를 끝까지 읽고 그 문서를 최우선 실행 계약으로 사용한다.

## 절차

1. 저장소의 인계 문서, 마이페이지 `page.tsx`, 레이아웃, 메뉴 컴포넌트, 인증·주문·환불 API, 관련 테스트를 읽는다.
2. 실제 페이지 파일에서 가능한 URL 목록을 만든다.
3. 코드 전체의 `href`, `Link`, `router.push`, `redirect`, URL hash를 수집해 페이지 목록과 대조한다.
4. 기본 계약 `/mypage`, `/mypage/orders`, `/mypage/profile`과 현재 구현의 차이를 표로 만든다.
5. 앱을 실행하고 브라우저에서 비로그인과 가능한 로그인 상태를 검수한다. 이 프로젝트에서는 활성화된 경우 `/qa-login`과 `website/.env.local`의 QA 관리자 설정을 사용하되 비밀값을 출력하지 않는다. 정적 분석만으로 실제 UX를 통과 처리하지 않는다.
6. 각 메뉴에 대해 클릭, 직접 주소, 새로고침, 뒤로가기, 앞으로가기, 데스크톱, 390px 모바일을 반복한다.
7. 프로필 API 생산자와 UI 소비자, 주문·환불 상태 생산자와 화면 레이블을 양쪽에서 함께 읽는다.
8. P0~P3 기준으로 발견 사항을 정리하고 역할 문서의 출력 경로에 보고서를 쓴다.
9. 테스트 종료 후 로그아웃하고 브라우저의 QA 세션을 폐기한다.

## 강제 판정 규칙

- 메뉴 클릭 후 pathname이 그대로이고 hash만 바뀌면 `P1`이다.
- 다른 메뉴의 주 콘텐츠가 동일 화면의 위나 아래에 남으면 `P1`이다.
- 직접 하위 URL이나 새로고침으로 같은 목적 화면을 재현할 수 없으면 `P1`이다.
- 뒤로가기·앞으로가기가 메뉴 방문 기록을 복원하지 못하면 `P2` 이상이다.
- 실제 브라우저 검증을 못 했으면 전체 판정은 `BLOCK`이다.
- 제품 코드 수정 없이 보고서만 작성한다. 수정 요청은 메인 에이전트가 별도 구현 단계로 다룬다.

## 출력

역할 문서의 보고서 템플릿과 완료 조건을 그대로 따른다. 통과 항목도 근거를 적고, 미검증 항목을 통과 항목에 섞지 않는다.

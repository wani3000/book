# My Page QA Orchestrator

## 목적

독립된 QA 에이전트가 마이페이지를 증거 기반으로 감사하고, 수정 권한이 주어진 경우에만 구현과 재검수를 분리해 진행한다.

## 워크플로우

### Phase 1: 준비

1. `.codex/harness/agents/mypage-journey-qa.md`와 `.codex/skills/mypage-journey-qa/SKILL.md`를 읽는다.
2. 대상 저장소, 기준 URL, 승인된 라우트 계약, 변경 금지 범위를 정리한다.
3. `_workspace/mypage-qa/{run-id}/00_input.md`에 민감 정보를 제외한 입력을 기록한다.

### Phase 2: 독립 감사

`spawn_agent`로 QA 에이전트 하나를 생성한다. 프롬프트에 다음을 반드시 포함한다.

- 역할 원본 경로
- 대상 저장소와 기준 URL
- 읽기 전용이며 제품 코드를 수정하지 않는다는 제한
- 보고서 출력 경로
- 라우팅·인증·주문·환불·PDF 권한의 완료 조건
- 활성화된 경우 `/qa-login`을 사용하고 자격 증명은 `website/.env.local`에서만 읽되 출력하지 않는다는 규칙
- 다른 에이전트와 직접 통신하지 말고 메인에 요청한다는 규칙

메인은 에이전트가 실행되는 동안 현재 라우트 파일과 진입 링크 목록을 준비한다. 결과가 필요할 때만 대기한다.

### Phase 3: 판정

1. QA 보고서의 증거와 실제 파일을 대조한다.
2. 근거 없는 발견, 재현할 수 없는 발견, 미검증을 통과로 둔 항목을 반려한다.
3. P0~P2와 출시 차단 항목을 사용자에게 우선 보고한다.

### Phase 4: 조건부 구현

사용자가 수정 또는 개발까지 명시한 경우에만 구현 작업자를 호출한다.

- 기본 라우트 계약은 `/mypage`, `/mypage/orders`, `/mypage/profile`이다.
- 구현 작업자는 승인된 파일 범위만 수정한다.
- QA 에이전트와 구현 작업자는 같은 제품 파일을 동시에 수정하지 않는다.
- 배포는 사용자가 별도로 지시한 경우에만 수행한다.

### Phase 5: 회귀 검수

구현 후 새 QA 에이전트가 최초 보고서의 P0~P2와 인접 시나리오를 전부 다시 실행한다. 빌드 통과만으로 출시 승인하지 않는다.

## 실패 처리

- QA 실행 실패: 한 번 재시도하고 정적 분석 결과와 미검증 범위를 보고한다.
- 로그인 세션 부재: 비로그인·정적 검증을 완료하고 로그인 시나리오는 `BLOCKED`로 유지한다.
- 브라우저 접근 불가: 실제 UX 판정을 `PASS`로 올리지 않는다.
- 스펙 충돌: 사용자 지시를 우선하고 인계 문서 수정을 별도 후속 작업으로 제시한다.

## 산출물

- 입력: `_workspace/mypage-qa/{run-id}/00_input.md`
- QA 보고서: `website/audit/mypage-journey-qa-{YYYY-MM-DD}/REPORT.md`
- 구현 요약이 승인된 경우: `_workspace/mypage-qa/{run-id}/02_implementation.md`
- 회귀 결과: `_workspace/mypage-qa/{run-id}/03_regression.md`

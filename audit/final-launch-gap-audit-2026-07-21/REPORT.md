# PHILIP BOOKS 최종 출시 기능 감사

- 감사일: 2026-07-21
- 대상: `https://codex-solo-builder-book.wani3000.chatgpt.site`
- 범위: 홈, 3개 상품 상세, Google 로그인, 결제, 내 서재/PDF, 환불, 관리자, 정책, 모바일, 보안·운영·SEO
- 증거: 이 폴더의 `01`~`10` PNG와 현재 저장소 소스, 비로그인 HTTP 응답

## 결론

현재 상태는 **콘텐츠·상세 페이지 공개에는 적합하지만 유료 판매의 정식 출시에는 부적합**하다. 가장 큰 이유는 PDF 세 파일이 `public/library-assets`에 있어 로그인·구매 여부와 관계없이 직접 URL로 열리기 때문이다. 이 상태에서는 환불 승인 후 권한을 회수해도 사용자가 기존 PDF 주소로 계속 읽을 수 있다.

또한 카카오페이·네이버페이 운영 키가 없어 실제 결제를 검증할 수 없고, 저장된 최신 환불 기능 버전은 아직 공개 사이트에 배포되지 않았다. 아래 P0을 모두 해결한 뒤 정식 판매를 시작해야 한다.

## P0 — 출시 전에 반드시 해결

### 1. PDF를 비공개 저장소로 이전

- 세 PDF를 `public/`에서 제거한다.
- Cloudflare R2 같은 비공개 객체 저장소에 둔다.
- `/api/library/[product]`가 로그인, 유효한 유료 주문, 미환불 상태를 확인한 뒤 파일을 스트리밍하거나 짧은 만료시간의 서명 URL을 발급하도록 바꾼다.
- `Cache-Control: private, no-store`, `X-Robots-Tag: noindex, noarchive`, 적절한 `Content-Disposition`을 실제 PDF 응답에 적용한다.
- 환불 완료·회원 정지·권한 회수 직후 접근이 차단되는지 E2E 테스트한다.
- 현재 원인: `app/library/catalog.ts`가 공개 경로를 보관하고, `app/api/library/[product]/route.ts`가 그 공개 경로로 리다이렉트한다.

### 2. 결제 운영 계약과 실제 결제 완주

- 카카오페이와 네이버페이 운영 심사·계약을 완료하고 운영 키와 사업자 연락처를 등록한다.
- 성공, 사용자 취소, 실패, 중복 콜백, 승인 후 브라우저 종료를 각각 실결제 또는 제공사 샌드박스로 검증한다.
- 결제 성공 뒤 주문, 마이페이지 결제내역, 내 서재 권한, 영수증/결제번호가 모두 일치하는지 확인한다.
- 현재 공개 사이트에서는 구매가 비활성이고 결제 운영 환경변수도 없다.

### 3. 결제 승인-DB 저장 불일치 복구

- 제공사 승인 성공 후 DB 주문 저장에 실패하면 사용자는 결제됐는데 책을 못 받는 상태가 된다.
- 결제 시도별 멱등 키를 강제하고, 제공사 거래 조회 → 주문 재생성/권한 복구가 가능한 관리자 화면 또는 자동 복구 작업을 만든다.
- `approving` 또는 `reconcile` 상태 장기 체류 알림과 재처리 버튼을 둔다.
- 카카오·네이버 콜백 모두 이 복구 흐름을 가져야 한다.

### 4. 환불 보류·실패 상태를 끝낼 관리 기능

- 네이버페이 `CancelNotComplete` 같은 비동기 환불을 주기적으로 재조회한다.
- `refund_pending`, `refund_review`, `refund_processing`에서 멈춘 주문을 조회·재시도·수동 완료·거절할 수 있어야 한다.
- 관리자의 모든 처리에는 담당자, 시각, 이전/이후 상태, 사유를 감사 로그로 남긴다.
- 실제 환불 완료가 확인된 뒤에만 주문을 `refunded`로 만들고 PDF 권한을 회수한다.

### 5. 동일 책 중복 결제 방지

- 결제 준비 API에서 같은 회원이 이미 해당 책의 `paid` 주문을 갖고 있는지 확인한다.
- 보유 중이면 새 결제를 만들지 말고 “이미 구매한 책입니다”와 내 서재 링크를 보여준다.
- 동시에 두 번 누르기, 새로고침, 콜백 재전송에도 주문이 하나만 생기도록 DB 제약/멱등 처리를 둔다.

### 6. 테스트 구매자 우회 권한을 운영 설정으로 제한

- 코드에 기본값으로 박힌 `oxaz1234@gmail.com`을 제거한다.
- 운영 환경의 명시적 허용 목록에 있을 때만 테스트 권한을 부여하고 관리자 화면/로그에서 확인 가능하게 한다.
- 테스트 권한으로 읽은 이력도 기록하고, 필요 시 즉시 회수 가능하게 한다.

### 7. 회원 탈퇴·재가입·구매기록 정책 일치

- 현재 탈퇴 계정은 같은 Google 계정으로 다시 로그인하면 활성 상태로 복구되고 기존 주문도 연결될 수 있다.
- 진짜 삭제인지, 재가입 시 구매 복원인지 제품 정책을 확정한다.
- 법정 보관 대상인 거래 기록은 계정 개인정보와 분리해 최소 보관하고 개인정보처리방침에 기간과 목적을 명시한다.
- 화면 안내, API 동작, DB 보관이 같은 규칙을 따라야 한다.

### 8. 최신 버전 배포와 운영 DB 마이그레이션

- 공개 사이트의 `/api/admin/refunds`는 현재 404이므로 새 환불 기능이 아직 운영 배포되지 않았다.
- 단, 최신 저장 버전도 공개 PDF 문제를 포함하므로 그대로 배포하지 않는다.
- P0 수정 → 스테이징 데이터 마이그레이션 → E2E → 운영 백업 → 배포 순서로 진행한다.

## P1 — 출시 품질과 사고 대응에 필요한 기능

### 보안

- CSP, HSTS, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy` 응답 헤더를 설정한다.
- 로그인, 결제 준비, 리뷰, 환불 요청에 속도 제한과 봇/반복 요청 방어를 둔다.
- 상태 변경 API에 Origin/CSRF 검증을 추가한다.
- 관리자 기능은 재인증 또는 2단계 인증을 권장하고 관리자 이메일만으로 역할을 부여하는 구조를 정기 점검한다.
- 비밀키 회전 절차와 유출 대응 문서를 둔다.

### 구매자 경험

- 결제 완료 화면과 마이페이지에 주문번호, 상품, 결제수단, 결제일시, 금액, 상태, 영수증/문의 경로를 보여준다.
- 결제 완료·실패·환불 접수·승인·거절 이메일을 보낸다.
- 미결제/권한 없음/환불됨/서비스 오류에 각각 다른 안내와 복구 행동을 제공한다.
- 결제 버튼 로딩 중 이중 클릭을 막고, 실패 후 안전하게 재시도할 수 있게 한다.

### 후기

- 임의의 주문번호 입력 대신 로그인한 회원의 실제 주문과 자동 연결한다.
- 주문 하나당 상품별 후기 1개, 수정/삭제 정책, 스팸·도배 제한을 둔다.
- 구매 확인 후기에만 “구매 인증”을 표시한다.
- 운영자 숨김·복구와 사유 기록을 남긴다.

### 관리자

- 주문, 결제 시도, 권한, 환불, 후기, 회원 상태를 한 화면에서 연결해 추적한다.
- 거래번호로 제공사 상태를 재조회하고 불일치를 고치는 기능을 둔다.
- 페이지네이션, 검색, 필터, CSV 내보내기와 변경 감사 로그를 추가한다.
- 결제 실패율, 권한 지급 실패, 환불 장기 대기, API 오류 급증 알림을 설정한다.

### 운영 안정성

- 오류 수집, 구조화 로그, 상태 확인 엔드포인트, 배포 후 스모크 테스트를 둔다.
- D1 정기 백업/내보내기, 복원 리허설, 마이그레이션 롤백 절차를 문서화한다.
- 카카오/네이버 장애 시 판매 중단 배너와 고객 공지 절차를 준비한다.
- 프로덕션과 스테이징의 키·DB·콜백 URL을 분리한다.

### 정책·고객지원

- 사업자명, 대표자, 사업자번호, 통신판매업 신고번호, 주소, 이메일, 전화번호를 최종 확인한다.
- 디지털 콘텐츠 청약철회 제한은 구매 전 체크와 명시적 동의를 받고 동의 시각·문구 버전을 저장한다.
- 환불 사유/상세정보의 수집 목적과 보관 기간을 개인정보처리방침에 반영한다.
- 주문이 자동 첨부되는 1:1 문의 채널과 응답 기준을 마련한다.

### 접근성

- 본문 바로가기 링크를 추가한다.
- 모바일 메뉴에 포커스 트랩, 배경 `inert`, 닫기 후 원래 버튼으로 포커스 복귀를 구현한다.
- 5초 자동 프로모션은 일시정지 버튼을 제공하고 `prefers-reduced-motion`에서는 자동 전환도 멈춘다.
- 비활성 구매 버튼은 색만으로 표현하지 말고 비활성 이유와 심사 완료 알림/문의 같은 다음 행동을 붙인다.
- 키보드, VoiceOver, 확대 200%, 대비를 실제 기기에서 별도 검사한다.

### SEO·측정

- 각 상품에 `Book`/`Product`/`Offer`, 브레드크럼, 실제 FAQ가 있을 때만 FAQ 구조화 데이터를 추가한다.
- Google Search Console과 네이버 서치어드바이저 소유권·사이트맵 제출을 확인한다.
- 상세 보기 → 로그인 → 결제 시작 → 결제 성공 → 첫 열람 → 환불의 전환 이벤트를 익명화해 측정한다.
- 분석/광고 쿠키를 쓸 경우 동의 UI와 정책을 맞춘다.
- PG 신뢰와 검색 자산 축적을 위해 소유한 커스텀 도메인을 권장한다.

## P2 — 출시 후 전환·운영 효율 개선

- 모바일 가로 카드에 스크롤 스냅과 위치 표시를 추가한다.
- 책 묶음 상품, 할인코드, 선물하기, 찜은 실제 수요가 생긴 뒤 추가한다.
- 환불 거절 뒤 새로운 증빙이 있을 때 재문의할 수 있는 지원 흐름을 둔다.
- 관리자 정산 대조 보고서와 월별 판매·환불 지표를 추가한다.
- 기기/세션 관리, 열람 기록 상세는 개인정보 최소수집 원칙 안에서 필요할 때만 추가한다.

## 기술 점검 메모

- 비로그인 `/api/library/codex`는 401이라 API 자체 인증은 작동한다. 그러나 공개 정적 PDF 주소가 이를 완전히 우회한다.
- 루트 응답에서 핵심 보안 헤더가 확인되지 않았다.
- `npm audit --omit=dev` 결과는 high/critical 없이 moderate 2건이다. Next에 포함된 PostCSS 문제이며 강제 수정은 큰 다운그레이드를 제안하므로 그대로 실행하지 말고 호환 업데이트를 추적한다.
- 현재 실제 PG 키가 없어 실결제, 결제취소, 카드/머니 잔액, 제공사 장애 상황은 검증하지 못했다.
- 스크린샷만으로 WCAG 준수, 스크린리더 읽기 순서, 키보드 포커스 전체를 확정할 수 없다.

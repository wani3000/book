# 다니엘의 노트 운영 런북

## 출시 환경 분리

- 운영과 로컬·스테이징은 OAuth 클라이언트, D1 데이터베이스, PG 키와 콜백 URL을 공유하지 않는다.
- 비밀값은 Sites 환경변수에만 저장하고 Git, 인계 문서, 브라우저 캡처에 남기지 않는다.
- `QA_LOGIN_ENABLED`와 `QA_LOGIN_ALLOW_PRODUCTION`은 공개 QA가 끝나는 즉시 `false`로 돌린다.

## 배포 전후 점검

1. `npm run lint && npm test`를 통과한다.
2. 배포 아카이브의 `dist/.openai/drizzle`에 최신 migration이 있는지 확인한다.
3. 운영 D1을 내보내고 복구 가능한 위치와 생성 시각을 기록한다.
4. 배포 후 `/api/health`, 홈, 상품 3개, Google·카카오 로그인, 마이페이지 하위 URL을 확인한다.
5. 결제가 활성화된 경우 승인·취소·실패 1건씩과 주문/권한 일치를 확인한다.
6. 오류가 늘면 결제 버튼을 환경변수로 끄고 직전 Sites 버전으로 롤백한다.

## 모니터링과 장애 공지

- `/api/health`가 503이거나 Worker 오류가 급증하면 신규 결제를 즉시 중단한다.
- `reconcile`, `approving`, `refund_pending`, `refund_processing` 상태가 10분 이상 유지되는지 매일 확인한다.
- 결제사업자 장애 시 홈과 상품 페이지에 판매 중단 안내를 게시하고 카카오톡 상담 채널에 동일 문구를 공지한다.
- 주문번호, 제공사 거래번호, 요청 ID를 기준으로 로그를 찾되 인증 토큰과 개인정보 원문은 기록하지 않는다.

## D1 백업과 복원

- 정식 판매 중에는 매일 D1 내보내기를 수행하고 30일 보관한다.
- 월 1회 별도 스테이징 DB에 최근 백업을 복원해 회원·주문·권한·환불 건수를 대조한다.
- migration은 순방향 적용을 원칙으로 하며, 파괴적 변경은 새 테이블 생성 → 복사 → 검증 → 전환 순서로 한다.
- 2026-07-22 로컬 D1에서 migration `0008_prelaunch_p1.sql` 적용 전 SQLite 백업을 만들고, 백업 복제본 복원 → migration 재적용 → `PRAGMA integrity_check`와 신규 테이블·열 검사를 통과했다. 이 결과는 운영 D1 백업을 대신하지 않는다.
- 운영 배포 후 `/api/health` 응답의 `schema`가 `0008`인지 확인한다. `migration-required`이면 판매를 열지 말고 운영 D1에 `0006`~`0008`을 순서대로 적용한다.

## 비밀키 회전과 유출 대응

1. Google/Kakao/PG/메일 키별 소유자와 마지막 교체일을 비공개 운영 장부에 기록한다.
2. OAuth·세션·PG 키는 90일마다, 담당자 변경 또는 유출 의심 시 즉시 교체한다.
3. 세션 비밀키 교체 시 기존 세션이 모두 종료됨을 공지한다.
4. 유출 시 해당 키 폐기 → 결제 중단 → 접근 로그 확인 → 새 키 배포 → 스모크 테스트 → 영향 사용자 통지 순서로 처리한다.

## 알림 메일

- `RESEND_API_KEY`, `TRANSACTIONAL_EMAIL_FROM`이 있으면 거래 알림을 전송한다.
- 키가 없거나 전송 실패한 알림은 `notification_outbox`에 남는다. 운영자는 `pending`·`failed`를 확인해 재처리한다.
- 카카오가 이메일을 제공하지 않은 계정에는 이메일을 보내지 않고 마이페이지 상태를 기준 정보로 사용한다.
- 발신 도메인 인증 후 결제 완료, 환불 신청, 환불 거절, 환불 완료를 각각 1건 테스트하고 수신함과 `notification_outbox.status='sent'`를 함께 확인한다.

## 카카오 계정 연결 해제

- Kakao Developers 앱의 관리자 키를 Sites 비밀 환경변수 `KAKAO_ADMIN_KEY`에 등록한다. 키 원문은 문서나 Git에 남기지 않는다.
- 회원 탈퇴와 로그인 수단 연결 해제 시 카카오 사용자 연결 해제 API가 먼저 성공해야 로컬 계정 처리를 계속한다.
- 테스트 계정에 Google과 Kakao를 모두 연결한 뒤 Kakao만 해제하고, Kakao Developers 사용자 목록과 `auth_identities`에서 모두 사라졌는지 확인한다.
- 회원 탈퇴는 처리 중인 환불이 없어야 하며, 성공 후 세션 종료·개인정보 익명화·카카오 연결 해제를 함께 확인한다.

## 관리자 보안

- 관리자 계정은 개인별 Google/Kakao 계정으로만 사용하고 공유 계정을 금지한다.
- 관리자 지정 이메일과 역할을 월 1회 점검하고 불필요한 권한은 즉시 회수한다.
- 회원·후기·환불·결제 변경은 `audit_logs`에 담당자, 대상, 시각, 사유를 남긴다.
- 회원·후기·결제·환불을 변경하는 관리 작업은 로그인 후 30분 이내에만 허용한다. 시간이 지났다면 관리자 화면에서 다시 로그인한다.
- 운영 관리자 Google/Kakao 계정 자체의 2단계 인증을 필수로 켜고 복구 수단은 서비스 운영자 개인 계정과 분리해 보관한다.

## 고객지원 기준

- 주문 문의는 마이페이지 주문 내역의 `주문 문의`를 사용한다. 제목에 주문번호가 자동 포함된다.
- 영업일 기준 1일 이내 첫 답변, 환불·결제 장애는 4시간 이내 접수 확인을 목표로 한다.
- 환불 사유와 상세 내용은 분쟁 처리 목적으로 최대 3년 보관한 뒤 삭제한다.
